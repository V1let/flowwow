## 2. Описание модулей (назначение, входные данные, ожидаемый результат)

### 2.1 Модуль авторизации
**Назначение:** регистрация пользователей, вход, работа с JWT.  
**Входные данные:** `email`, `password`, `name`, `phone`.  
**Ожидаемый результат:** успешная регистрация/аутентификация, выдача JWT, корректные ошибки при неверных данных.

### 2.2 Модуль работы с БД
**Назначение:** чтение/запись сущностей (товары, категории, отзывы, пользователи, корзина).  
**Входные данные:** CRUD‑операции на сущностях, ключи/ID.  
**Ожидаемый результат:** консистентные данные, корректные ограничения (FK, уникальные поля).

### 2.3 Модуль админ‑панели
**Назначение:** управление товарами/категориями/заказами/отзывами/пользователями.  
**Входные данные:** формы админки, действия (создание, редактирование, удаление).  
**Ожидаемый результат:** корректные изменения в БД, информирование об ошибках.

### 2.4 Модуль поиска и фильтрации
**Назначение:** поиск товаров по параметрам, фильтрация по категории, цене, строке.  
**Входные данные:** `categoryId`, `minPrice`, `maxPrice`, `search`.  
**Ожидаемый результат:** корректный список товаров, соответствующий фильтрам.

### 2.5 Пользовательский интерфейс
**Назначение:** отображение каталога, карточек, отзывов, корзины, избранного.  
**Входные данные:** данные API, пользовательские действия.  
**Ожидаемый результат:** корректная отрисовка UI и состояний.

## 3. Тест‑кейсы (по одной функции на модуль)

### 3.1 Авторизация — регистрация
**ID:** AUTH‑REG‑01  
**Шаги:**
1. Отправить `POST /api/auth/register` с валидными `name/email/phone/password`.
2. Проверить ответ.
**Ожидаемый результат:** `200 OK`, в ответе `token`, `id`, `email`.

### 3.2 Работа с БД — создание категории
**ID:** DB‑CAT‑01  
**Шаги:**
1. Отправить `POST /api/admin/categories` с валидным `name`.
2. Запросить `GET /api/admin/categories`.
**Ожидаемый результат:** категория появляется в списке.

### 3.3 Админ‑панель — создание товара
**ID:** ADMIN‑PROD‑01  
**Шаги:**
1. Открыть `admin-product-edit.html`.
2. Заполнить форму, нажать «Сохранить».
**Ожидаемый результат:** товар создан, остаёмся на странице редактирования.

### 3.4 Поиск и фильтрация — фильтр по категории
**ID:** SEARCH‑CAT‑01  
**Шаги:**
1. Отправить `GET /api/products?categoryId=<id>`.
2. Проверить список.
**Ожидаемый результат:** товары только выбранной категории.

### 3.5 UI — карточка товара в каталоге
**ID:** UI‑CARD‑01  
**Шаги:**
1. Открыть `pages/catalog.html`.
2. Проверить карточки: фото, название, цена, кнопка.
**Ожидаемый результат:** карточки отображаются корректно, описание ограничено по высоте.

## 4. Проведение изолированного тестирования

Тест‑прогоны выполнены на локальном стенде с запущенным backend (`http://localhost:8080`).  
Для API‑тестов использовался админ JWT. Созданные тестовые сущности удалены после проверки.

**Результаты прогона:**
1. AUTH‑REG‑01 — **Pass**  
   Создан пользователь `modtest_20260220211144@example.com`, в ответе получены `token/id/email`.
2. DB‑CAT‑01 — **Pass**  
   Создана категория `MT Категория 20260220211144`, проверена в списке админ‑категорий.
3. ADMIN‑PROD‑01 — **Pass**  
   Создан товар `MT Букет 20260220211144`, id получен, после теста удалён.
4. SEARCH‑CAT‑01 — **Pass**  
   Запрос по `categoryId` возвращает товары только нужной категории (в тесте товаров не было, что допустимо).
5. UI‑CARD‑01 — **Pass**  
   Ручная проверка в браузере выполнена: карточки отображаются корректно.
